Project (.claude/) vs User (~/.claude/) — what lives where and why
PROJECT STRUCTURE — .claude/ (committed to git)
CLAUDE.md
LAYER 1 GUIDANCE Always-on project context. Loaded every turn. Describes project conventions, workflow rules, agent roles. Bypassable under goal pressure — not enforcement.
AGENTS.md
LAYER 1 GUIDANCE Industry-standard agent guidance (AAIF). Agent-agnostic — works with Claude, GPT, Gemini. Documents agent roles, responsibilities, expected behaviors.
.mcp.json
LAYER 2 MCP server configuration. Use ${VAR} refs only — no hardcoded secrets. Committed safely because credentials live in env vars.
.claude/settings.json
LAYER 2-3 ENFORCEMENT Permissions (allow/deny/ask rules) + hooks (PreToolUse, PostToolUse, SessionStart). The deterministic enforcement boundary. Cannot be bypassed by the reasoning loop.
.claude/rules/
LAYER 1 GUIDANCE Path-scoped instruction files (e.g. frontend.md, api.md). Lazy-loaded only when Claude enters matching path. Reduces token cost of CLAUDE.md.
.claude/skills/
On-demand workflow definitions. ~100 tok idle, full SKILL.md loaded on activation. Course skills: /think, /spec, /audit-aiuc1, /tool-select, /retro, /harness, /worktree.
.claude/agents/
Subagent definitions (YAML or MD). Each specifies: system prompt, allowed tools, model. Course agents: recon, analyzer, reporter, red-team.
.claude/hooks/
LAYER 3 ENFORCEMENT Shell scripts called by settings.json hooks. Must be chmod +x. Exit code 0 = allow, non-zero = block. Deterministic — always fires, no reasoning involved.
plans/
Spec packages and approved plans. Set plansDirectory in settings.json so plans land here instead of ~/.claude/plans/ (which uses random names).
harnesses/
Security Minion blueprints. Each harness has: blueprint.yaml (structure), fixed_steps/ (deterministic), flexible_steps/ (AI-driven). Produced by /harness.
reports/
Audit outputs (AUDIT-*.md), sprint retrospectives (RETRO-*.md). Produced by /audit-aiuc1 and /retro.
resources/
Reference materials: approach-selection.md, swarm-patterns.md. Read by /spec during planning.
USER STRUCTURE — ~/.claude/ (never committed)
~/.claude/CLAUDE.md
Personal conventions active in all projects. Your preferred communication style, personal shortcuts, cross-project preferences. Lower priority than project CLAUDE.md.
~/.claude/settings.json
Personal permission preferences. Project settings.json takes precedence for project-specific rules. Use for global deny rules you always want (e.g., never delete git history).
~/.claude/skills/
Personal skills available in every project. Good for general-purpose workflows not project-specific. Project skills take precedence on name collision.
~/.claude/agents/
Personal subagents available everywhere. Same precedence rules as skills.
~/.claude/plans/
Default plan location uses random names (piped-humming-wand.md style). Override with plansDirectory in project settings.json to save plans in your repo instead.
~/.claude/plugins/
Installed plugins and their cache. Managed via /config. Plugin state persists here between sessions.
~/.claude/MEMORY.md
Auto-generated persistent memory. Claude writes project conventions, user preferences, and feedback here during sessions. Loaded at conversation start. Index file — individual memories are separate .md files.
~/.claude.json
LAYER 4 Authentication state, API keys, session tokens. NEVER commit this file. Add to .gitignore. Lives at home root, not inside ~/.claude/.
CLAUDE.md @import — Modular Context
Break large CLAUDE.md files into modules using @import. Claude loads imported files on demand.
# CLAUDE.md (root) @import ./context/style-guide.md @import ./context/agent-roles.md @import ./context/security-rules.md # Only the root CLAUDE.md is always loaded. # Imported files load when referenced or when # Claude determines they're relevant.
Skills Frontmatter — context: fork | inherit
ModeWhat the subagent seesWhen to use
context: fork Copy of parent context at fork time. Parent changes after fork are invisible to subagent. Parallel workers. Isolation needed. Default for most course agents.
context: inherit Live shared context with parent. Both see the same conversation in real time. Sequential handoffs where subagent must see parent's latest output.
DEFENSE LAYER CLASSIFICATION — Inside vs Outside the Reasoning Loop

⚠ Inside Loop — Guidance (probabilistic, bypassable)

  • CLAUDE.md Agent reads this — but CAN ignore under goal pressure or context overflow
  • AGENTS.md Describes expected behavior — not enforced by runtime
  • .claude/rules/*.md Path-scoped instructions — lazy-loaded, still probabilistic
  • .claude/skills/ Activation depends on Claude choosing to invoke — not guaranteed
  • .claude/agents/*.md System prompts — agent may deviate under adversarial input
  • plans/*.md Intent documentation — not a runtime constraint
  • harnesses/*/flexible_steps/ The "AI part" — Claude generates these dynamically

✓ Outside Loop — Enforcement (deterministic, always fires)

  • settings.json → permissions.deny Tool calls blocked at OS level — CANNOT be bypassed by the model
  • settings.json → hooks.PreToolUse Hook runs BEFORE every tool call — can block or modify
  • settings.json → hooks.PostToolUse Hook runs AFTER every tool call — can audit or alert
  • settings.json → sandboxing OS-level process isolation — filesystem and network restricted
  • .claude/hooks/*.sh The actual enforcement code — deterministic shell scripts
  • --max-turns, --max-budget-usd Hard caps enforced by the harness, not the model
  • harnesses/*/fixed_steps/ Deterministic pipeline — runs regardless of model output
  • harnesses/*/blueprint.yaml Defines the enforcement boundary for the sprint
WEEK-BY-WEEK DIRECTORY POPULATION — What Gets Created When
Week File / Directory Created Purpose & Teaching Objective
Week 1CLAUDE.mdFirst project context file. Establish CCT framework, communication style, project conventions.
Week 1.claude/settings.jsonFirst permissions file. Add a simple deny rule (e.g., deny deleting .env). Introduce enforcement concept.
Week 2.mcp.jsonRegister first MCP server. Introduce tool access scope, ${VAR} pattern for secrets.
Week 2.claude/rules/security.mdPath-scoped security rules. Introduce lazy loading vs always-on context tradeoff.
Week 3.claude/skills/think.mdFirst course skill. Introduce skills frontmatter: name, description, context mode.
Week 4.claude/skills/spec.mdSecond skill. Spec generation workflow. Introduce plans/ directory and plansDirectory setting.
Week 5.claude/agents/analyzer.mdFirst subagent. Introduce agent YAML structure: system prompt, tools, model override.
Week 6.claude/hooks/pre-tool.shFirst enforcement hook. PreToolUse pattern — block or log tool calls before execution.
Week 7.claude/skills/audit-aiuc1.mdAIUC-1 audit skill. Introduce reports/ directory. First artifact-producing skill.
Week 8.claude/agents/recon.md, red-team.mdSecurity-focused subagents. Orchestrator-worker pattern for multi-agent attack/defend lab.
Week 9harnesses/scanner/blueprint.yamlFirst harness. Introduce fixed_steps/ vs flexible_steps/ boundary.
Week 10AGENTS.mdAdd AAIF-standard agent documentation. Introduce agent-agnostic conventions.
Week 11.claude/skills/harness.md, worktree.mdProduction skills. Worktree isolation for parallel agents.
Week 12CLAUDE.md @import refactorBreak monolithic CLAUDE.md into modules. Introduce @import syntax and lazy loading.
Week 13-15Capstone: full structureAll directories populated. Complete project with governance, enforcement, and audit trail.
.claude/settings.json — Configuration Template
Minimal starter (Week 1)
{ "plansDirectory": "plans", "permissions": { "deny": [ "Bash(rm -rf *)", "Bash(git push --force)" ] } }
Production (Week 9+)
{ "plansDirectory": "plans", "permissions": { "allow": ["Read", "Glob", "Grep"], "deny": [ "Bash(rm -rf *)", "Bash(curl * | sh)", "WebFetch(*.malicious.*)" ] }, "hooks": { "PreToolUse": [{ "matcher": "Bash", "hooks": [{"type":"command", "command":".claude/hooks/pre-tool.sh"}] }] } }
ARTIFACT FLOW — How Skills Read and Write the Directory Structure
/think ─────────────────────────────→ [conversation only — no artifact] │ reads: CLAUDE.md (CCT framework) │ ▼ /spec ──────────────────────────────→ plans/SPEC-{name}.md │ reads: resources/approach-selection.md, resources/swarm-patterns.md │ reads: .claude/agents/ (available worker definitions) │ ▼ Plan Mode (human reviews + approves) → plans/{plan-name}.md │ reads: plans/SPEC-*.md, CLAUDE.md, codebase [read-only] │ human edits via Ctrl+G before approving │ ▼ Build ──────────────────────────────→ source files, working tool │ uses: .claude/agents/ (delegates subtasks to workers) │ uses: .mcp.json (MCP tool access) │ constrained by: settings.json (deny rules block dangerous calls) │ constrained by: .claude/hooks/ (enforcement scripts run around every tool) │ ▼ /audit-aiuc1 ───────────────────────→ reports/AUDIT-{name}.md │ reads: working tool, AIUC-1 domains A-F │ ▼ /harness ───────────────────────────→ harnesses/{type}/blueprint.yaml │ harnesses/{type}/fixed_steps/ │ harnesses/{type}/flexible_steps/ │ reads: plans/SPEC-*.md │ ▼ /worktree ──────────────────────────→ ../worktrees/{name}-{n}/ (isolated copy) │ reads: .claude/agents/, settings.json │ each worktree gets its own .claude/ copy for isolation │ ▼ Sprint execution → /retro ──────────→ reports/RETRO-{sprint}.md reads: git log, sprint artifacts, harness outputs
