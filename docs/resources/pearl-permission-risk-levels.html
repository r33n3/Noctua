Stage 0 — Extends GLOE
Design, Scoping & Experimentation
Fail fast. Ideas enter, get prototyped rapidly, and either prove viability or get killed. No governance commitment — just basic safety rails and empirical evidence.
Context Package — Created Here

The initial context package is assembled during ideation: problem definition, candidate models, data requirements, compliance framework identification, and Engineering Assessment Stack analysis. This package travels with the use case through every subsequent stage — it IS the institutional knowledge about what was tested, what worked, and why decisions were made.

Guardrails — Lightweight
AI Gateway (basic): traffic logging, PII scanning, credential detection
Cost controls: per-session and per-day spending caps

Isolation: no production system access from sandbox
PeaRL: project in scoping state — no active findings

Gate: AI Use Case Contract
Evidence-based declaration — models, tools, agents, data classification, applicable compliance frameworks. Governance committee reviews. Approved → enter Development with defined scope.
Stage 1 — Aligns with GLOE Development
Development — Build to Contract
Build the real system per the use case contract. PeaRL active governance generates findings. Deviations from contract require quick approval updates. Context package evolves with implementation.
Context Package — Reworked & Deepened

The context package from Stage 0 is reworked through experimentation. Prompts are refined, evaluation datasets are built, model selections are validated against real workloads. Implementation decisions update the package — "we chose Sonnet because Haiku failed at X" becomes documented institutional knowledge. Deviations from the original contract are recorded as approved amendments with justification.

PeaRL Gate 1: Dev → Preproduction
Contract fulfilled? All findings resolved or reviewer-excepted? CI/CD security validation passed? Context receipts attested? Models and tools match declarations?
Stage 2 — Aligns with GLOE Preproduction
Preproduction — Validate & Harden
Contract is locked. What was declared is what gets tested. Adversarial testing, guardrail validation, and multi-layered security verification. No deviations permitted.
Validation Activities
Adversarial testing (red team) against declared models
Guardrail validation against declared tools
Behavioral testing against declared agents
Multi-layered testing framework (unit, integration, e2e)
AI gateway hardened — full content moderation
Deep observability, tracing, and audit trails
Governance — Full Review
PeaRL: Reviewer oversight — all exceptions must be decided
AIUC-1: Domains A, B, D full depth + C, F validation
Behavioral: AGP-01–05 anomaly detection active
Pipeline: automated compliance verification
⬥
PeaRL Gate 2: Preproduction → Production
All AIUC-1 domains validated? Red team results reviewed? Behavioral monitoring confirmed active? Guardrails held under adversarial testing? Kill switch tested?
Stage 3 — Aligns with GLOE Production
Production — Deploy & Govern Continuously
Deploy what was validated. Monitor against contract expectations. Reassess on triggers: model changes, incidents, regulation changes, or periodic review schedule.
Continuous Operations
Performance monitoring & drift detection
Feedback loops driving improvement cycles
Shadow testing before deploying updates
Cost tracking, optimization, and allocation tagging
Incident response procedures
Rollback strategies for degraded performance
Governance — Architectural Enforcement
PeaRL: JWT-only auth — no local-mode flags permitted
AIUC-1: All 6 domains at full depth — continuous compliance
Gateway: rate limiting, anomaly alerting, content moderation
Isolation: separate OS users, secrets in Vault, containers
Reassessment: quarterly (Tier 2) or triggered by change
